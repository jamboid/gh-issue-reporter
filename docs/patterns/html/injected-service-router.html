Pattern

Injected Service Router

The Express router accepts a service object as a parameter instead of constructing one — making every HTTP status code testable in milliseconds with a mock, and making the underlying service swappable without touching the route.

Layer · src/githubRouter.js Pattern · Dependency Injection Consumer · Host Express app

Problem statement

An Express route handler needs to create a GitHub issue. The direct approach imports the GitHub client inside the route handler, passes in credentials, and calls it immediately. This feels natural but creates two problems.

First, the route is untestable without real credentials and a live network. There's no way to verify that the route returns 401 when GitHub rejects the token, or 422 when the issue body is malformed, without actually triggering those conditions against GitHub's API. Tests become slow, flaky, and environment-dependent.

Second, the route is coupled to one specific implementation. If you want to add a caching layer, swap GitHub for a different issue tracker, or log requests before posting, you have to edit the route file. The route conflates "handle HTTP" with "talk to GitHub."

Core insight: The router handles HTTP mechanics (extract fields, map errors to status codes, return JSON). The service handles API mechanics. By passing the service in as a parameter, the connection between them is made by the caller — and can be swapped for a mock in tests.

How it works

githubRouter is a factory that takes a service and returns a configured Express Router. In production the caller passes a real githubService. In tests the caller passes a mock object — the router never knows the difference.

Implementation in this codebase

githubRouter never imports or constructs a service. It accepts whatever object is passed in and calls service.createIssue(). The minimal expected interface is: an object with a createIssue({ title, body, label }) method that returns a promise.

  src/githubRouter.js 
import { Router } from 'express'

export function githubRouter(service) {
  const router = Router()

  router.post('/issues', async (req, res) => {
    const { title, body, label } = req.body
    try {
      const result = await service.createIssue({ title, body, label })
      res.json(result)
    } catch (e) {
      res.status(e.status || 500).json({ error: e.message })
    }
  })

  return router
}
  Host Express app 
import { githubRouter, githubService } from '@jamboid/gh-issue-reporter/server'

// Caller constructs the service and hands it to the router.
const service = githubService({
  repo: process.env.GITHUB_REPO,
  token: process.env.GITHUB_TOKEN,
})

app.use('/api/github', githubRouter(service))

// To swap in a caching layer or a different tracker,
// just pass a different object — the router file doesn't change.
  tests/githubRouter.test.js 
const mockService = { createIssue: vi.fn() }
const app = express()
  .use(express.json())
  .use('/', githubRouter(mockService))

// Test a 401 without a real GitHub token.
mockService.createIssue.mockRejectedValue(
  Object.assign(new Error('Not authorized'), { status: 401 })
)
const res = await request(app).post('/issues').send({})
expect(res.status).toBe(401)

// No network call. No credentials. Milliseconds.
Minimal interface: The router only requires service.createIssue({ title, body, label }) that returns a promise resolving to { url }. Any object satisfying that shape — githubService, a mock, or a custom wrapper — is a valid service.

Trade-offs

+ Advantages

  • Full test coverage without a network. Every HTTP status code the router can emit is testable by controlling what the mock service throws or returns. Tests run in milliseconds.
  • Router stays focused on HTTP. It extracts fields from the request, delegates to the service, and maps results back to HTTP responses. It does nothing else — no credentials, no API knowledge.
  • Service is swappable. A custom service that logs to a database before posting, rate-limits requests, or routes to a different tracker replaces githubService without touching the router.

− Disadvantages

  • Informal service interface. There's no TypeScript interface enforcing { createIssue(...): Promise<{ url }> }. A service that returns { html_url } silently breaks the response without a runtime error.
  • Wiring is the caller's responsibility. The caller must know to construct the service before the router, pass it in, and mount the router at the right path. There's no default configuration if they omit it.

Key files

  • JS
    src/githubRouter.js
    Accepts a service as a parameter; handles HTTP request/response only. Never imports or constructs the service itself. Exported via the server entry point.
  • JS
    src/githubService.js
    The default service implementation. Satisfies the router's expected interface: an object with createIssue({ title, body, label }) that returns Promise<{ url }>.
  • TST
    tests/githubRouter.test.js
    Demonstrates testing the router with a mock service. Covers every HTTP status code path — 401, 422, 500 — without a network or credentials. Shows the pattern paying off immediately.
