Pattern

Submission Adapter Seam

A required submit prop on the component is the only point of connection between the form and the outside world — swap the function to change how issues are submitted without touching the component.

Layer · src/IssueReporter.vue Pattern · Prop Injection / Seam Consumer · Host app

Problem statement

A bug-reporting component needs to create GitHub issues. The obvious approach puts the GitHub API call directly inside the component — but this creates three problems that compound over time.

First, the token gets exposed. Client-side components are bundled into the browser. A GitHub token passed as a prop ends up readable by anyone who opens DevTools. Projects that can't tolerate this must route the call through a server — but if the component hardcodes the direct path, there's no way to do that without forking it.

Second, the component is untestable in isolation. Every test has to either mock the network or use real credentials. You can't test the form, validation, spinner, or success/error states without also setting up a GitHub API stub at the network level.

Third, swapping submission targets requires editing the component. If a project already has a backend or wants to route to a different issue tracker, someone has to reach inside IssueReporter.vue and change it. There's no configuration option — just a code change.

Core insight: The seam is the submit prop — a single required function that the component calls with form data. Whatever that function does is the caller's concern. The component just fires it and handles the result.

How it works

The component never decides how to submit. It exposes a submit prop — a function — and calls it with the form data when the user submits. The caller chooses which function to pass. Two built-in adapters ship with the package; any async function matching the signature works in their place.

Implementation in this codebase

The component treats submit as an opaque function. It extracts form data, calls props.submit(...), and handles whatever comes back. It has no knowledge of GitHub, fetch, or any specific backend.

  src/IssueReporter.vue 
const props = defineProps({
  submit: { type: Function, required: true },
  // ...
})

async function handleSubmit() {
  submitting.value = true
  error.value = null
  try {
    const result = await props.submit({
      title: form.title,
      description: form.description,
      type: form.type,
      context: contextText.value,
    })
    issueUrl.value = result.url
  } catch (e) {
    error.value = e.message
  } finally {
    submitting.value = false
  }
}
  Host app — direct adapter (token in browser env) 
import IssueReporter, { createDirectAdapter } from '@jamboid/gh-issue-reporter'

// Factory captures repo and token. Component receives only the function.
const submit = createDirectAdapter({
  repo: 'owner/repo',
  token: import.meta.env.VITE_GITHUB_TOKEN
})
  Host app — proxy adapter (token stays on server) 
import IssueReporter, { createProxyAdapter } from '@jamboid/gh-issue-reporter'

// POSTs form data to your server; the server calls GitHub with the token.
// No credentials in the browser at all.
const submit = createProxyAdapter() // defaults to POST /api/github/issues
  Host app — custom adapter (any target) 
// Any async function matching the signature works.
// No component changes needed — just swap the function.
const submit = async ({ title, description, type, context }) => {
  const res = await fetch('/my/endpoint', {
    method: 'POST',
    body: JSON.stringify({ title, description, type, context }),
  })
  const data = await res.json()
  if (!res.ok) throw new Error(data.message)
  return { url: data.url }
}
Adapter contract: Any function satisfying ({ title, description, type, context }) => Promise<{ url }> is a valid adapter. The component never cares what's inside — it only reads result.url on success or e.message on failure.

Trade-offs

+ Advantages

  • Credentials stay where they belong. Direct adapters keep the token in the calling app's environment. Proxy adapters keep it on the server entirely. The component never sees it.
  • Testable without network calls. Pass a vi.fn() as submit and assert on spinner, error, and success states without touching the network or needing credentials.
  • No fork for custom targets. Any async function matching the signature works. Projects routing to Jira, Slack, or a different GitHub org don't touch the component.

− Disadvantages

  • Caller must choose an adapter. submit is a required prop with no in-component fallback. Omitting it is a Vue runtime warning, not a clear install-time error.
  • Informal adapter contract. There's no TypeScript interface enforcing the signature. A custom adapter that returns { html_url } instead of { url } silently breaks the success state.
  • Error handling is the adapter's job. An adapter that catches and swallows errors leaves the UI in a perpetual submitting state — the component has no way to detect this.

Key files

  • VUE
    src/IssueReporter.vue
    Accepts submit as a required prop and calls it in handleSubmit(). Renders spinner, inline error, and success link based on the result. Has no knowledge of any specific backend.
  • JS
    src/adapters.js
    createDirectAdapter calls the GitHub API directly from the browser using a token captured at factory time. createProxyAdapter POSTs to a server-side route, keeping the token out of the browser entirely.
  • JS
    src/index.js
    Re-exports the component and both adapters as the package's public client-side surface. Consumers import whatever combination they need from one entry point.
