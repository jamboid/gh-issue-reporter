Patterns

Architectural patterns used across this package — each written up with the problem it solves, how it's wired in this project, and the trade-offs that come with it.

Vue 3 · Composition API Express Vitest Dual-env package

These six patterns together describe how gh-issue-reporter stays testable, secure, and configurable across two runtime environments. The entry points enforce a hard client/server boundary; the adapter seam and factory pattern keep credentials out of components; the injected service router makes the Express layer independently testable; the shared API primitive gives GitHub calls one home; and SubmissionError gives failures a consistent shape from the API layer through to the UI.

Each page describes the problem, shows the implementation in this codebase, and lists the advantages and disadvantages so you can judge when — and when not — to reach for the pattern.

Dual Entry Points

dual-entry-points.html
package.json exports map client + server

The exports field in package.json defines two named entry points: "." for browser code and "./server" for Node.js code. Frontend bundlers cannot see Express; the server entry cannot see Vue. Each environment gets exactly what it needs and nothing it doesn't.

Factory-Configured Adapter

factory-configured-adapter.html
src/adapters.js factory / closure sealed credentials

A factory takes configuration (repo, token) and returns the adapter — sealing credentials in a closure at startup. The returned function takes only form data. The component receives a plain function and never sees credentials, Vue DevTools included.

Submission Adapter Seam

submission-adapter-seam.html
IssueReporter.vue submit prop swappable

The component's submit prop is the seam — the point where submission behaviour can be changed without touching the component. Two adapters ship with the package; any async function satisfying ({ title, description, type, context }) => Promise<{ url }> works in their place.

Injected Service Router

injected-service-router.html
githubRouter.js dependency injection testable HTTP

githubRouter is a factory that accepts a service object and returns an Express Router. It never constructs the service itself. Tests pass a mock object; production passes githubService. Every HTTP status code the router can emit is testable without a network call.

Shared API Primitive

shared-api-primitive.html
githubService.js single source of truth no duplication

postGitHubIssue is the single function that knows how to talk to the GitHub Issues API — headers, serialisation, response parsing, and error shape. Both the client-side adapter and the server-side service delegate to it; neither duplicates the fetch logic.

Typed Error with HTTP Status

submission-error.html
SubmissionError Error subclass .status + .message

SubmissionError extends the native Error class and adds a .status property. Every throw site in the GitHub API layer uses it. The router reads .status to set the HTTP response code; the component reads .message to display inline.
