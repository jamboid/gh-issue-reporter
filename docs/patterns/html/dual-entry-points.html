Pattern

Dual Entry Points

Two named export paths in package.json keep browser code and server code in entirely separate bundles — a frontend bundler cannot accidentally pull in Express.

Layer · package.json Pattern · Dual Entry Points Environments · Browser + Node.js

Problem statement

A package that ships both a Vue component and an Express router has a fundamental conflict: Vue components run in the browser, Express runs in Node.js. The two environments have almost nothing in common.

If all exports live behind a single entry point, a frontend bundler processing that import has to at least consider everything the package re-exports. Express depends on Node.js built-ins — http, net, stream — that don't exist in the browser. The bundler either errors out or produces a broken bundle.

Bundle size is a second problem. A client app that only needs the Vue component shouldn't pay the cost of shipping Express and all its dependencies to the browser. Tree-shaking can't reliably help: some bundlers can't safely shake Node.js-specific code, and the attempt itself can introduce subtle breakage.

Core insight: Two named export paths in package.json give the bundler hard guarantees. Importing the client entry provably cannot reach the server entry. No tree-shaking heuristics required.

How it works

The exports field in package.json acts as a router: it maps import specifiers to files on disk. "." routes to the client file; "./server" routes to the server file. Each file re-exports only what belongs in that environment.

Implementation in this codebase

The exports field in package.json is the source of truth. Two named paths, two re-export files, zero overlap between them.

  package.json 
{
  "name": "@jamboid/gh-issue-reporter",
  "exports": {
    ".": "./src/index.js",
    "./server": "./src/server.js"
  }
}
  src/index.js — client entry 
// Only browser-safe exports here — no Node.js modules.
export { default } from './IssueReporter.vue'
export { createDirectAdapter, createProxyAdapter } from './adapters.js'
  src/server.js — server entry 
// Only Node.js / Express exports here — no Vue, no browser globals.
export { githubService } from './githubService.js'
export { githubRouter } from './githubRouter.js'
  Consumer usage 
// In a Vue app — Vite bundles this; Express is unreachable.
import IssueReporter, { createDirectAdapter } from '@jamboid/gh-issue-reporter'

// In an Express app — Node.js; Vue is never required.
import { githubRouter, githubService } from '@jamboid/gh-issue-reporter/server'

// Importing the wrong entry in the wrong environment fails loudly,
// not silently — which is exactly what you want.
Bundler support: Vite, webpack, and Rollup all resolve the exports field. Importing from '@jamboid/gh-issue-reporter' is guaranteed by the package contract — not a naming convention — to never include src/server.js.

Trade-offs

+ Advantages

  • Clean environment separation. Frontend bundlers process only browser-safe code; Node.js processes only server code. No cross-environment pollution by construction.
  • Bundle safety. A Vite build that imports from '@jamboid/gh-issue-reporter' cannot reach Express, even if Express is a dependency of the same monorepo.
  • Explicit public API. The two entry files are the package's public surface. A developer reading src/index.js sees the entire client API in three lines.

− Disadvantages

  • Two APIs to document and maintain. Changes to the package's public surface must be considered for both entry points — it's easy to accidentally expose something in one that shouldn't be there.
  • Non-obvious server import path. '@jamboid/gh-issue-reporter/server' is a subpath that developers must know to look for. It won't appear in IDE autocomplete unless the editor supports package exports resolution.

Key files

  • {}
    package.json
    Defines the exports map — the source of truth for what the package exposes and from which path. Changing the map here immediately affects what consumers can import.
  • JS
    src/index.js
    Client entry point. Re-exports IssueReporter as default, and createDirectAdapter, createProxyAdapter as named exports. No server-side imports.
  • JS
    src/server.js
    Server entry point. Re-exports githubService and githubRouter. No Vue imports. This is the file a Node.js app receives when it imports from '@jamboid/gh-issue-reporter/server'.
