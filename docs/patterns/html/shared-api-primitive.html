Pattern

Shared API Primitive

One function owns every detail of the GitHub API call — URL, headers, error shape, response normalisation — so two callers in different environments get identical behaviour from a single source of truth.

Layer · src/githubService.js Pattern · Shared Primitive / DRY Callers · adapters.js + githubService

Problem statement

When multiple parts of a codebase need to call the same external API, the natural instinct is to write the API call wherever it's first needed, then copy it when a second caller appears. This produces a fragile codebase in predictable ways.

Headers drift. The GitHub API requires specific headers — Authorization, Accept, X-GitHub-Api-Version. Written in two places, one copy will eventually fall behind: wrong API version, missing header, subtly different error handling.

Response parsing duplicates. Both callers need to read html_url from the response JSON and normalise it to { url }. Two implementations means two places where this mapping can be wrong.

Error handling splits. One caller might check res.ok before parsing JSON; another might read the error body for a user-facing message. The difference is invisible until something breaks in production.

Core insight: The primitive is a function that knows everything about talking to the GitHub Issues API. The thin wrappers above it know only what differs between their callers — how to format the body, what labels to use. If GitHub changes their API version header, there is exactly one line to update.

How it works

Two callers sit above the primitive: createDirectAdapter formats the body from form fields; githubService.createIssue accepts a pre-formatted body from the router. Neither touches headers, status codes, or response shapes. Those live exclusively in postGitHubIssue.

Implementation in this codebase

postGitHubIssue owns the entire GitHub API interaction. The two thin wrappers prepare their inputs and delegate — neither touches fetch directly, neither constructs the Authorization header, neither reads html_url.

  src/githubService.js — the primitive 
export async function postGitHubIssue({ repo, token, title, body, labels }) {
  const res = await fetch(`https://api.github.com/repos/${repo}/issues`, {
    method: 'POST',
    headers: {
      Authorization: `Bearer ${token}`,
      'Content-Type': 'application/json',
      Accept: 'application/vnd.github+json',
      'X-GitHub-Api-Version': '2022-11-28',
    },
    body: JSON.stringify({ title, body, labels }),
  })
  const data = await res.json()
  if (!res.ok) throw new SubmissionError(res.status, data.message || `GitHub API error: ${res.status}`)
  return { url: data.html_url }
}
  src/adapters.js — thin client wrapper 
// Only concern: format the body from form fields, then delegate.
export function createDirectAdapter({ repo, token }) {
  return ({ title, description, type, context }) => {
    return postGitHubIssue({
      repo, token, title,
      body: formatIssueBody({ description, context }),
      labels: [type],
    })
  }
}

// No fetch. No headers. No html_url. Just the glue.
  src/githubService.js — thin server wrapper 
// Only concern: accept a pre-formatted body from the router, then delegate.
export function githubService({ repo, token }) {
  if (!token) throw new SubmissionError(503, 'token is required')
  return {
    createIssue({ title, body, label }) {
      return postGitHubIssue({ repo, token, title, body, labels: [label] })
    },
  }
}

// No fetch. No Accept header. No html_url. Just the glue.
The deletion test: If you deleted postGitHubIssue, the complexity wouldn't disappear — it would reappear in both wrappers. Every line you'd have to duplicate is a line the primitive is earning its keep. That's the signal that the abstraction is the right size.

Trade-offs

+ Advantages

  • Locality. All GitHub API mechanics — URL construction, headers, response parsing, error shape — live in one function. One place to read, debug, and update.
  • Consistent error shape. Both callers automatically throw SubmissionError with the correct HTTP status. There's no way for them to diverge in how failures are reported.
  • Tests collapse. The primitive's tests cover the fetch logic once. Wrapper tests only verify their own glue — no need to re-stub the same fetch headers in two test files.

− Disadvantages

  • One more indirection to trace. A developer reading createDirectAdapter for the first time hits an import and must follow it to understand what actually happens at the network level.
  • The primitive's interface must satisfy both callers. Both pass labels as an array, body as a pre-formatted string. If a future caller needs different behaviour, the primitive's interface must grow or a second primitive must be created.

Key files

  • JS
    src/githubService.js — postGitHubIssue
    The primitive. Owns the GitHub API URL, all required headers, JSON serialisation, res.ok checking, error shape, and html_url → url normalisation. Only function that calls fetch for GitHub.
  • JS
    src/adapters.js — createDirectAdapter
    Thin client-side wrapper. Formats the issue body from form fields using formatIssueBody, then delegates everything else to postGitHubIssue. No fetch, no headers.
  • JS
    src/githubService.js — githubService
    Thin server-side wrapper. Accepts a pre-formatted body from the router and delegates to postGitHubIssue. Guards against a missing token at construction time.
