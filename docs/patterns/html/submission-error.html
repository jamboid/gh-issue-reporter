Pattern

SubmissionError

A typed Error subclass that carries both an HTTP status code and a human-readable message — giving the Express router and the Vue component exactly what they each need from a single thrown object.

Layer · src/githubService.js Pattern · Typed Error / Error Subclass Consumers · githubRouter + IssueReporter

Problem statement

When something goes wrong during issue submission, two different parts of the system need information about that failure: the Express router needs an HTTP status code to forward to the client, and the Vue component needs a human-readable message to display inline.

Without a consistent error shape, each throw site makes its own decision. One might throw a plain object { status: 422, message: '...' }. Another might throw a standard Error with .message but no .status. A third might throw a string. The router then has to guess: res.status(e.status || 500) — the || 500 is a silent fallback for "I don't know what this is."

Plain object throws have a further problem: they aren't Error instances. They have no stack trace, don't work correctly with instanceof, and behave unexpectedly with some promise rejection handlers and logging tools that expect real Error objects.

Core insight: One typed class — SubmissionError extends Error — gives the failure a name, a consistent shape, and real Error semantics. Every throw site in the GitHub API layer produces the same object. Callers don't need to guess what they caught.

How it works

Two throw sites produce SubmissionError. Two catch sites consume it. Each reads the property it cares about — the router reads .status, the component reads .message — from the same object.

SubmissionError Throw sites (GitHub API layer) githubRouter (reads .status) IssueReporter (reads .message)

Implementation in this codebase

SubmissionError extends the native Error class and adds a .status property. Every throw site in the GitHub API layer uses it — and callers get exactly what they need from the single caught object.

  src/githubService.js 
export class SubmissionError extends Error {
  constructor(status, message) {
    super(message)
    this.status = status
  }
}

// Real Error semantics: stack trace, instanceof, promise rejection handlers.
// Callers get .status (for HTTP) and .message (for display) from one object.
  src/githubService.js — throw sites 
// When the GitHub API returns a non-2xx response:
if (!res.ok) throw new SubmissionError(
  res.status,
  data.message || `GitHub API error: ${res.status}`
)

// When githubService is constructed without a token:
if (!token) throw new SubmissionError(503, 'token is required')

// Construction-time failure: loud, typed, carries a meaningful status code.
// Not buried inside the first request — caught immediately at startup.
  src/githubRouter.js — catch site 
} catch (e) {
  // .status is always present on SubmissionError.
  // || 500 is a defensive fallback for unexpected throws from custom services.
  res.status(e.status || 500).json({ error: e.message })
}
  src/IssueReporter.vue — catch site 
} catch (e) {
  // .message is always present — it's an Error property.
  // The component doesn't need .status; it only shows text to the user.
  error.value = e.message
}
Both callers get what they need from the same thrown object. The router reads .status to build the HTTP response. The component reads .message to populate the inline error. Neither has to massage the error into a different shape first.

Trade-offs

+ Advantages

  • One error shape everywhere. Every throw in the GitHub API layer produces a SubmissionError. Callers don't need to check whether .status exists or whether the thrown value is an Error or a plain object.
  • Full Error semantics. Stack traces, instanceof checks, and promise rejection handlers all work correctly. A logging tool that expects Error instances won't drop the error silently.
  • Construction-time failures are explicit. githubService throws a SubmissionError(503, ...) immediately if the token is missing — not on the first user request.
  • Tests can assert on both properties. rejects.toMatchObject({ status: 422 }) works because toMatchObject checks properties regardless of class. Stricter tests can use instanceof SubmissionError.

− Disadvantages

  • SubmissionError is not part of the public API. It's exported from src/githubService.js but not re-exported from either entry point. A custom adapter author who wants to throw the same type must import from an internal path.
  • The router still duck-types .status. The e.status || 500 pattern works for SubmissionError but also for any Error with .status attached. The router doesn't enforce that errors are SubmissionError instances.

Key files

  • JS
    src/githubService.js
    Defines SubmissionError. Both postGitHubIssue (API errors) and githubService constructor (missing token) throw it. This is the only file that creates SubmissionError instances.
  • JS
    src/githubRouter.js
    Reads e.status and e.message from caught errors to build the HTTP response. The || 500 fallback handles unexpected throws from custom services.
  • VUE
    src/IssueReporter.vue
    Reads e.message from caught adapter errors to populate the inline error display. Doesn't need .status — only the human-readable text matters in the UI.
  • TST
    tests/githubService.test.js
    Verifies that SubmissionError carries both .status and .message. Uses rejects.toMatchObject({ status: 422 }) to assert on the typed shape.
